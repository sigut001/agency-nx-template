# 🌍 08 - i18n & Multi-Language Concept

Dieses Dokument beschreibt das Konzept zur Skalierung von Webprojekten für verschiedene Sprachen, basierend auf Inhalten aus Firebase.

## Zentrale Sprachsteuerung

Die Auswahl der verfügbaren Sprachen (z.B. `de`, `en`, `fr`) erfolgt zentral in einer `languages.config.ts`. Diese Liste steuert:

1. Die UI der Sprachumschaltung.
2. Den Build-Prozess (Prerendering-Pfade).
3. Die Validierung der CMS-Inhalte.

## Firebase CMS Struktur

Wir haben zwei Wege zur Auswahl der Datenhaltung in Firestore evaluiert:

1. **Feld-basierte Übersetzung**: Ein Dokument enthält `title_de`, `title_en`. (Wird bei vielen Sprachen unübersichtlich).
2. **Dokument-Duplikats-Modell (Favorit)**: Jedes Dokument existiert pro Sprache in einer eigenen Struktur oder mit einem Sprach-Flag.
   - _Vorteil_: Wenn eine Sprache nachträglich hinzugefügt wird, können einfach neue Dokumente "geklont" werden, ohne das Schema der bestehenden Dokumente zu ändern.
   - _Struktur_: `collections/{lang}/pages/{pageId}` oder `pages/{pageId}/translations/{lang}`.

## Prerendering & SEO

Damit Crawler die Inhalte finden:

- Die Build-Pipeline generiert für jede Sprache ein eigenes Unterverzeichnis in der Distribution: `dist/de/`, `dist/en/`.
- Firebase Hosting leitet Anfragen basierend auf der URL oder Headern entsprechend weiter.
- **Hreflang Tags**: Werden automatisiert in den `<head>` injiziert, um Google die Sprachvarianten mitzuteilen.

## Offene Fragen

- **Initial-Setup**: Wie automatisieren wir das Duplizieren der Dokumente in Firebase, wenn eine neue Sprache aktiviert wird?
- **Fallbacks**: Was passiert, wenn ein Inhalt für eine Sprache noch nicht übersetzt wurde? (Show original / Show empty / Redirect).
- **i18n Implementation**: Brauchen wir eine Library wie `react-i18next` für UI-Strings (Buttons, Label), wenn der Haupt-Content ohnehin aus Firebase kommt? Oder ziehen wir _alles_ (auch UI-Labels) aus dem CMS?
